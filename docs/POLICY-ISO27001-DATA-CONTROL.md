## 2. การจำแนกชั้นข้อมูล (A.5.12)

|---|---|---|
| ลับ (Confidential) | เอกสารหลักฐาน compliance, ข้อมูลติดต่อลูกค้า, ผล audit | RLS จำกัดเฉพาะสมาชิกธุรกิจ + เข้ารหัส |

## 3. การควบคุมการเข้าถึง (A.5.15, A.5.18, A.8.2)
- **Row Level Security (RLS) ทุกตาราง** — ผู้ใช้เห็นเฉพาะข้อมูล "ธุรกิจ" ที่ตนเป็นสมาชิก
  (`is_org_member` / `is_member`) บังคับที่ชั้นฐานข้อมูล ปลอมจากฝั่ง client ไม่ได้
- แบ่งบทบาท: owner / admin / auditor / member — auditor อ่านอย่างเดียวเพื่อการตรวจ ISO
- บัญชีผู้ดูแลระบบใช้ MFA และปฏิบัติตาม `SOP-ACCESS-CONTROL-SSO.md` (ขอสิทธิ์–อนุมัติ–ถอนสิทธิ์)
- Secrets แยกตาม environment (ดู Environment Map ใน CLAUDE.md) — ห้ามใช้ secret ข้ามระบบ

## 4. การเข้ารหัส (A.8.24)
- ข้อมูลระหว่างส่ง: TLS 1.2+ ทุกช่องทาง (HTTPS/WSS)
- ข้อมูลจัดเก็บ: เข้ารหัส at-rest โดยโครงสร้างพื้นฐาน Supabase/AWS (AES-256)
- ไม่จัดเก็บรหัสผ่านแบบอ่านได้ — Supabase Auth ใช้ bcrypt

## 5. การสำรองข้อมูล (A.8.13)
- Schema ทั้งหมดเก็บเป็น migration ใน Git (กู้โครงสร้างได้ทุกเมื่อ)
- ข้อมูล production: daily backup ตามแผน Supabase Pro (อยู่ระหว่างยกระดับ — ความเสี่ยง R5)
  ระหว่างนี้ทำ SQL export รายสัปดาห์เก็บใน repository ส่วนตัว
- ทดสอบ restore อย่างน้อยปีละ 1 ครั้ง และบันทึกผล

## 6. การบันทึกและติดตาม (A.8.15, A.8.16)
- Login logs: Supabase Auth Logs · การเข้าถึงโครงสร้างพื้นฐาน: Cloudflare Audit Log, GitHub Security Log
- ตาราง `validations` เก็บ audit trail การตรวจข้อกำหนด: ใครตรวจ ผลอะไร อ้างมาตรฐานเวอร์ชันไหน เมื่อไหร่
- เหตุการณ์ระบบ (signup/activate/upgrade) บันทึกใน `marketing_events` และ GA4

## 8. การเก็บรักษาและการลบข้อมูล (A.5.33, A.8.10)
- ลูกค้าลบข้อมูลได้ด้วยตนเองทุกเมื่อ: ลบ workspace (ล้างข้อมูลธุรกิจทั้งหมดผ่าน FK cascade)
  หรือลบบัญชีถาวรผ่านฟังก์ชัน `delete-account` (ลบถึง auth.users) — มีจริงในระบบ ใช้งานได้แล้ว
- ข้อมูลที่ลบแล้วไม่สามารถกู้คืน และหลุดจาก backup ตามรอบ retention ของ backup
