| วันที่ | 2026-07-07 |
| ทบทวน | ทุก 12 เดือน + หลังเกิดเหตุจริง |

## วัตถุประสงค์
รักษาความมั่นคงปลอดภัยและความพร้อมใช้ของบริการระหว่างเหตุขัดข้อง (5.29) และเตรียม ICT ให้พร้อมต่อความต่อเนื่องธุรกิจ (5.30)

## เป้าหมายการกู้คืน (ค่าเริ่มต้น — ผู้บริหารยืนยัน)
- **RTO** (เวลาที่ยอมให้ระบบล่ม): ≤ 24 ชม. สำหรับบริการหลัก
- **RPO** (ข้อมูลที่ยอมให้สูญ): ≤ 24 ชม. (สอดคล้อง daily backup — ต้องเปิดจริง ดู [[R5]])

## สถานการณ์และแนวทางรับมือ

| สถานการณ์ | ผลกระทบ | แนวทางกู้คืน |
|---|---|---|
| **Cloud provider ล่ม** (Supabase/Cloudflare) | บริการหยุด | พึ่ง redundancy ระดับ platform (8.14) · ติดตาม status page · สื่อสารผู้ใช้ · ไม่มี failover ข้าม provider (ยอมรับความเสี่ยงตามขนาดธุรกิจ) |
| **โน้ตบุ๊กเครื่องเดียวหาย/เสีย** | หยุดพัฒนา/ดูแล + เสี่ยงข้อมูล | source code กู้จาก GitHub · secrets กู้จาก Bitwarden ([[R18]]) · เครื่องเข้ารหัส BitLocker กันข้อมูลรั่ว · **single point of failure — ควรมีแผนเข้าถึงสำรอง** |
| **คีย์รั่ว/ถูกยึด** | ใช้บริการด้วยเงินเรา/เข้าข้อมูล | rotate ทันที (มีขั้นตอนแล้ว [[R1]]) · revoke เก่า · ตรวจ usage · ทำ incident report |
| **ข้อมูล DB สูญ/เสียหาย** | ข้อมูลลูกค้าหาย | restore จาก platform backup (Pro — ต้อง verify R5) + app-level export · **ต้องทดสอบ restore ≥2/ปี** (O3) |
| **บัญชีผู้ดูแลถูกยึด** | เข้าถึงทุก workspace | MFA (R3) เป็นด่านป้องกัน · recovery codes เก็บใน Bitwarden |

## ข้อจำกัดที่ยอมรับ (Residual — small team)
- ผู้ปฏิบัติงานคนเดียว = **ไม่มีผู้สำรองเข้าถึงระบบทันที** ถ้าเจ้าของไม่พร้อม → ควรมีแผนมอบสิทธิ์ฉุกเฉิน/ผู้สืบทอดเมื่อธุรกิจโต
- ไม่มี multi-cloud failover — ยอมรับตามขนาดธุรกิจ ณ ปัจจุบัน (เจ้าของความเสี่ยงอนุมัติ)
