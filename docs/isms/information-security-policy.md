## 1. วัตถุประสงค์
ธุรกิจมุ่งมั่นปกป้อง **ความลับ (Confidentiality) ความถูกต้องครบถ้วน (Integrity) และความพร้อมใช้งาน (Availability)** ของข้อมูลสารสนเทศของธุรกิจและของลูกค้า เพื่อสร้างความเชื่อมั่น ปฏิบัติตามกฎหมาย (PDPA) และรองรับการเติบโตของแพลตฟอร์ม

## 2. ขอบเขต
ครอบคลุมระบบ บุคลากร กระบวนการ และผู้ให้บริการภายนอกตามที่ระบุใน [README.md](README.md) (ข้อ 4.3)

## 3. หลักการ (Policy Statements)
1. **การเข้าถึงตามสิทธิ์ที่จำเป็น (least privilege)** — ผู้ใช้/ระบบเข้าถึงเฉพาะข้อมูลที่จำเป็น บังคับผ่าน RLS และการแบ่งบทบาท (control 5.15, 8.2, 8.3)
2. **ยืนยันตัวตนที่มั่นคง** — บัญชีที่มีสิทธิ์สูง (แอดมิน) ต้องเปิด **MFA** (control 8.5)
3. **การจัดการคีย์ลับ** — ความลับทั้งหมด (API keys, credentials) เก็บใน secret manager นอก repo, หมุนตามรอบ, ห้าม commit เข้า git (control 8.24, 5.10)
4. **พัฒนาอย่างปลอดภัย** — ตรวจ secret scanning + dependency audit ทุก push, review ก่อน merge (control 8.25, 8.28, 8.8)
5. **สำรองและกู้คืน** — ข้อมูลลูกค้าต้องมี backup และทดสอบการกู้คืนตามรอบ (control 8.13, 5.30)
6. **บันทึกและเฝ้าติดตาม** — เปิด log ของระบบสำคัญและทบทวนเป็นรอบ (control 8.15, 8.16)
7. **ความเป็นส่วนตัว** — ประมวลผล PII ตาม PDPA และมีข้อตกลงกับผู้ให้บริการ (control 5.34, 5.22)
8. **บริหารตามความเสี่ยง** — เลือกมาตรการจากผลประเมินความเสี่ยง บันทึกใน [SoA](statement-of-applicability.md) (ข้อ 6.1.3)
9. **รายงานเหตุการณ์** — เหตุการณ์/ช่องโหว่ต้องรายงานตาม [SECURITY.md](../../SECURITY.md) (control 6.8)

## 4. บทบาทและความรับผิดชอบ (ข้อ 5.3)
| บทบาท | ความรับผิดชอบ |
|---|---|
| ผู้บริหาร | อนุมัตินโยบาย, จัดสรรทรัพยากร, ทบทวนฝ่ายบริหาร (9.3) |
| IT Security | ประเมินความเสี่ยง, ดูแล controls, ตอบสนองเหตุการณ์ |
| ผู้พัฒนา | secure coding, ไม่ commit ความลับ, แก้ช่องโหว่ตามกำหนด |
| ผู้ใช้ทุกคน | ปฏิบัติตามนโยบาย, รายงานเหตุการณ์ |

## 5. การปฏิบัติตามและบทลงโทษ
การไม่ปฏิบัติตามนโยบายอาจนำไปสู่กระบวนการทางวินัย (control 6.4) ผู้บริหารทบทวนความสอดคล้องอย่างน้อยปีละครั้ง
