## 2. สถานะ action จากการทบทวนครั้งก่อน — (9.3.2 a)
**ไม่มี** — นี่คือ Management Review **ครั้งแรก** ของธุรกิจ (ปิด NC เชิงประวัติในตัว: ก่อนหน้าไม่เคยมี MR)

## 3. การเปลี่ยนบริบท + ผู้มีส่วนได้เสีย — (9.3.2 b, c)
- **ภายใน:** สร้าง dev project แยก (แต่ schema ยังไม่ apply — NC-01); แก้ production identity ที่เคยเข้าใจผิด ([[R11]]/[[R19]]); ลบ rogue edge functions + rotate คีย์ ([[R20]]); ยัง single-operator ([[R18]])
- **ภายนอก:** ยัง **pre-launch — ยังไม่มีผู้ใช้จริงลงทะเบียน** (`auth.users` ว่าง ณ ตรวจ R13) → ผลกระทบต่อลูกค้าจริงยังเป็นศูนย์ แต่ต้องปิดช่องโหว่ก่อนเปิดตัว; PDPA/พ.ร.บ.คอมพิวเตอร์ยังบังคับใช้
- **ผู้มีส่วนได้เสีย:** ยังไม่มีการเปลี่ยนความคาดหวังที่มีนัยสำคัญ (ยังไม่มีลูกค้า/สัญญา)

### 4.3 ผล Internal Audit (d.3)
อ้าง [audit report 1/2569](audits/2569/internal-audit-2026-07-08.md) — headline **NC-01: 8.31/R15 ถูกบันทึกว่าปิดแต่ dev schema ไม่เคย apply จริง** (public schema ว่าง — verified live) = บทเรียนซ้ำของ config/identity drift

### 4.4 การบรรลุวัตถุประสงค์โดยรวม (d.4)
โครงสร้าง ISMS + control เชิงเทคนิคหลัก **บรรลุ**; ช่องว่างที่เหลือเป็น **operational** (MFA, backup verify, DPA, awareness) ที่มีแผน+กำหนดเวลาแล้ว

## 5. Feedback จากผู้มีส่วนได้เสีย — (9.3.2 e)
**ยังไม่มี** (pre-launch ไม่มีลูกค้า/ข้อร้องเรียน/incident จากภายนอก) — ให้เปิดช่องรับ feedback ([SECURITY.md](../../SECURITY.md), support@b-tctraining.com) พร้อมเปิดตัว

## 6. ผลประเมินความเสี่ยง + สถานะ treatment — (9.3.2 f)
- 20 ความเสี่ยง: ปิด 15 · เปิด 5 · **NC-01 → R15 กลับเป็นเปิด** (dev/prod ยังไม่แยกจริง)
- Residual risk ที่เสนอให้ผู้บริหาร**ยอมรับชั่วคราว** (มี compensating + แผนปิด): R3(MFA), R4(DPA), R5(backup), R17(disposal) — ทั้งหมดความเสี่ยงต่ำ-กลาง มีกำหนดปิด
- SoA: 🔴=0 (หลังแก้ 8.31 กลับ 🟡 จาก NC-01 → SoA 🔴 ยังคง 0 แต่ 8.31 = 🟡 รอ verify)

## 7. โอกาสปรับปรุงต่อเนื่อง — (9.3.2 g)
- OFI-01: ตั้งช่องทาง **verify prod ที่ทำซ้ำได้** (prod เข้าผ่าน MCP ไม่ได้) — สำคัญต่อ audit รอบหน้า
- OFI-02: ระบุ `sck-crm` เป็นนอกขอบเขต ISMS ใน environment-map
- OFI-03: automate การเก็บ metric 9.1 (cron รัน security_checks.sql + log)
