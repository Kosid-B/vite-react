## บริบท
เขียนขึ้นหลังเหตุการณ์จริง (R13, 2026-07-05) ที่ฟีเจอร์ทีม/สมาชิกบน production พังจากการแก้ไขสิทธิ์ที่ไม่ครบวงจร
เหตุการณ์นั้นถูกพบ+แก้ไขได้ผลดี แต่ **ทำแบบ ad-hoc ไม่มีขั้นตอนที่เขียนไว้ล่วงหน้า** — เอกสารนี้บันทึกขั้นตอนให้ทำซ้ำได้

## 1. การเตรียมพร้อม (5.24 — Planning and preparation)
- ช่องทางรายงานเหตุการณ์: `support@b-tctraining.com` (ดู [SECURITY.md](../../SECURITY.md))
- เครื่องมือตรวจสอบ: Supabase MCP (`get_advisors`, `get_logs`, `execute_sql` แบบ read-only), Cloudflare dashboard logs
- ผู้รับผิดชอบ: IT Security ประเมิน+แก้ไขทางเทคนิค, ผู้บริหารตัดสินใจเรื่องมีผลกระทบทางธุรกิจ/กฎหมาย

## 2. ขั้นตอนการรับมือ (5.25–5.26 — Assessment, decision, response)

```
1. รับรู้เหตุการณ์ (จากผู้ใช้แจ้ง / advisor เตือน / สังเกตจากการตรวจสอบ)
   ↓
2. ประเมินระดับความรุนแรง (ดูตารางด้านล่าง) — ห้ามลงมือแก้ก่อนเข้าใจ scope
   ↓
3. ตรวจสอบแบบ READ-ONLY ก่อนเสมอ (execute_sql อ่านอย่างเดียว, get_advisors, get_logs)
   — ห้ามเขียน/แก้ไขจนกว่าจะเข้าใจสาเหตุที่แท้จริง (root cause)
   ↓
4. เตรียมวิธีแก้ + SQL/โค้ดที่ชัดเจน พร้อมอธิบายผลกระทบ
   ↓
5. ขอ confirm จากเจ้าของระบบก่อนแก้ไข production เสมอ (ยกเว้นได้รับอนุญาตล่วงหน้าเป็นการทั่วไป)
   ↓
6. ลงมือแก้ไข → verify ทันทีด้วย query ตรวจสอบ (ห้ามเชื่อว่า "รันสำเร็จ" = "ผลลัพธ์ถูกต้อง")
   ↓
7. บันทึกเหตุการณ์ในทะเบียนความเสี่ยง (risk-register.md) พร้อมวันที่ + หลักฐาน
```

### ระดับความรุนแรง (Severity)
| ระดับ | คำอธิบาย | เวลาตอบสนอง | ตัวอย่าง |
|---|---|---|---|
| **วิกฤต (Critical)** | กระทบผู้ใช้ทั้งหมด / ข้อมูลรั่ว / เข้าระบบไม่ได้ | ทันที | คีย์รั่วที่ยังใช้งานอยู่, RLS bypass ที่เห็นข้อมูลข้ามผู้เช่า |
| **สูง (High)** | ฟีเจอร์หลักใช้งานไม่ได้ / ช่องโหว่ที่ยังไม่ถูกใช้ | ภายในวันเดียว | เหตุการณ์ R13 (ฟีเจอร์ทีม/สมาชิกพัง) |
| **กลาง (Medium)** | ฟีเจอร์รองกระทบ / ความเสี่ยงต่ำ ผลกระทบจำกัด | ภายในสัปดาห์ | R14 (bucket listing เปิดกว้างเกินจำเป็น) |
| **ต่ำ (Low)** | ไม่กระทบผู้ใช้ /ปรับปรุงเชิงป้องกัน | ตามรอบพัฒนาปกติ | ปรับปรุงเอกสาร/policy |

## 3. การเก็บหลักฐาน (5.28 — Collection of evidence)
สำหรับทุกเหตุการณ์ระดับ**กลางขึ้นไป** ต้องบันทึก:
- วันที่/เวลาที่พบ + วิธีที่พบ (ใครแจ้ง/เครื่องมือไหนเตือน)
- Query/คำสั่งที่ใช้ตรวจสอบ (read-only) + ผลลัพธ์ที่ได้
- Query/คำสั่งที่ใช้แก้ไข + ผลลัพธ์ verify หลังแก้
- ผลกระทบจริง (ใครโดนกระทบ, นานแค่ไหน, ข้อมูลอะไรบ้าง)
- เก็บไว้ใน [risk-register.md](risk-register.md) เป็นแถวเดียวต่อเหตุการณ์ (รูปแบบ R-number)

## 4. การเรียนรู้จากเหตุการณ์ (5.27 — Learning from incidents)
หลังแก้ไขเสร็จทุกครั้ง ต้องตอบคำถามนี้ในทะเบียนความเสี่ยงหรือเอกสารที่เกี่ยวข้อง:
1. **สาเหตุที่แท้จริง (root cause)** คืออะไร — ไม่ใช่แค่ "อาการ" ที่เห็น
2. **ทำไมไม่ถูกจับได้เร็วกว่านี้** — ขาด monitoring/alert อะไรหรือไม่ (ดู control 8.16)
3. **ป้องกันไม่ให้เกิดซ้ำได้อย่างไร** — เพิ่ม control/CI check ใหม่ไหม
4. ตัวอย่างจริง: R11 (config drift) มี postmortem เต็มใน [environment-map.md](environment-map.md) §0

## 5. การรายงาน (6.8 / control ที่เกี่ยวข้อง)
- เหตุการณ์ระดับ**สูงขึ้นไป** ต้องแจ้งผู้บริหารทันทีที่พบ (ไม่ต้องรอแก้เสร็จ)
- เหตุการณ์ที่กระทบข้อมูลส่วนบุคคลของลูกค้า (PII) ต้องประเมินภาระผูกพันตาม PDPA แยก (อาจต้องแจ้งเจ้าของข้อมูล/หน่วยงานกำกับ)
